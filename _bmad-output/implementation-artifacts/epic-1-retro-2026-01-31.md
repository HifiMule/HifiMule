# Retrospective: Epic 1 - Foundation & Project Genesis

**Date:** 2026-01-31
**Project:** JellyfinSync
**Epic:** 1 - Foundation & Project Genesis

## Summary
Epic 1 established the core multi-process architecture, the background daemon with system tray integration, and the detachable Tauri-based UI skeleton.

## Successes (What Went Well)
- **Extreme Memory Efficiency:** The `jellysync-daemon` idle footprint is ~937 KB, well under the 10MB goal.
- **Architectural De-coupling:** Successfully isolated the sync engine from the UI runtime, ensuring background stability.
- **Vibrant Hub Aesthetics:** The Shoelace-based UI skeleton provides a modern, premium feel from the start.
- **Cross-Platform Foundation:** Verified tray functionality and workspace build on Windows/macOS.

## Challenges (What Was Difficult)
- **Complex Build Chain:** The Cargo -> Tauri -> npm/Vite dependency chain led to 'program not found' issues and environment path complexities.
- **macOS Main Thread Constraint:** Implementing the system tray required careful threading to satisfy macOS requirements without blocking the daemon's async logic.
- **Dependency Bloat:** Early discovery of `image` crate overhead necessitated feature-flag optimization to maintain low memory usage.

## Lessons Learned
- **Audit Dependencies Early:** Removing unused features in core crates is essential for performance-first utilities.
- **Environment Parity is Key:** Windows PowerShell path issues highlighted the need for robust developer environment documentation.
- **Thread Safety vs. OS Constraints:** Cross-platform tray implementations require a hybrid threading approach (main thread for UI, background for logic).
